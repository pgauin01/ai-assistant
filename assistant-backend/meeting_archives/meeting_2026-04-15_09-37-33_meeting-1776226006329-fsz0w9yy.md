### 1. The Elevator Pitch

Document ingestion for RAG systems is where most implementations fail because naive approaches treat it like simple text extraction. The reality is that poor chunking destroys semantic relationships, causing your retrieval to return irrelevant context that confuses the LLM. For my advanced RAG chatbot, I engineered a multi-stage ingestion pipeline that balanced semantic preservation with retrieval precision. The business impact was significant—by implementing hybrid retrieval with intelligent chunking and metadata tagging, I improved answer accuracy by 40% and enabled citation tracking that users could verify, dramatically increasing trust in the system.

### 2. Core Mechanics

**Parsing Layer:** I used PyPDFLoader to extract text while preserving structural metadata like page numbers and section headers. This wasn't just about getting text out—I needed to maintain document hierarchy so chunks could be tagged with their source location for later citation.

**Chunking Strategy:** The critical decision was chunk size and overlap. I implemented RecursiveCharacterTextSplitter with 500-token chunks and 50-token overlap. That overlap is crucial—when a concept spans chunk boundaries, the overlap acts as a semantic bridge ensuring the embedding model captures the full relationship. Without it, you get context fragmentation where related ideas are artificially separated.

**Hybrid Indexing:** I built a dual-retrieval system. Dense retrieval using BGE-small-en-v1.5 embeddings stored in FAISS handles semantic similarity—understanding that "machine learning model" relates to "neural network training." But dense embeddings miss exact keyword matches like specific API names or error codes. So I added BM25 for sparse keyword matching. I fused both using Reciprocal Rank Fusion, which merges rankings without needing score normalization.

**Metadata Enrichment:** Every chunk got tagged with source filename, page number, and section headers. This enabled retrieval filtering by document type and citation tracking—the chatbot could say "According to page 12, section 3.2..."

### 3. Top Options & Trade-offs

**Fixed vs. Semantic Chunking:** Fixed-size chunking (my 500-token approach) is simple and fast but can split sentences mid-thought. Semantic chunking using sentence transformers respects natural boundaries but is computationally expensive and can create wildly varying chunk sizes that hurt retrieval consistency. **Decision rule:** Use fixed chunking with overlap for technical docs where speed matters; use semantic chunking for narrative content where preserving complete thoughts is critical.

**Dense-only vs. Hybrid Retrieval:** Pure dense retrieval (just FAISS) is simpler to implement and works well for conceptual queries. But it fails on exact matches—searching for "error code 404" might return semantically similar but wrong error codes. Hybrid retrieval adds complexity and latency but handles both semantic and keyword queries. **Decision rule:** If your domain has precise terminology (legal, medical, technical), hybrid is mandatory. For general knowledge, dense-only suffices.

**Chunk Size Selection:** Smaller chunks (200 tokens) give precise retrieval but lose context. Larger chunks (1000+ tokens) preserve context but reduce precision and hit token limits faster. **Decision rule:** Test empirically with your data, but 400-600 tokens with 10% overlap is a solid starting point for most technical documentation.
