Technical Deep Dive

### assistant
### 1. The Elevator Pitch  
We’re looking at a self‑hosted Azure DevOps agent that’s timing out during the React build. The fix is a mix of resource tuning, build caching, and pipeline configuration tweaks—no fancy new tooling needed.

### 2. Core Mechanics  
When you run `npm run build` or `yarn build`, webpack spins up a Node process that compiles all your JSX, bundles, and minifies. That process can consume 4–8 GB of RAM and 100 % CPU for several minutes, especially with a large component tree. If the agent’s VM is a small size or the disk is slow, the Node process will start swapping or hit I/O limits, causing the Azure pipeline to kill it after the timeout. Azure DevOps also enforces a default 60‑minute timeout per job, so any build that takes longer will fail.

### 3. Top Options & Trade‑offs  
- **Scale the agent VM**: Move from a B2s to a D4s or higher. More cores and RAM mean webpack can run in parallel and stay in memory. The trade‑off is higher hourly cost and a bit more maintenance (you’re still managing the OS).  
- **Enable build caching**: Use Azure Pipelines caching or a self‑hosted cache server to persist `node_modules` and the `build/` folder. This cuts the build time by 70–80 % on repeat runs. The downside is a bit of extra pipeline complexity and the need to invalidate the cache when dependencies change.  
- **Adjust pipeline timeout**: Increase the job timeout to 90 or 120 minutes. That’s a quick win if the build is just borderline. The risk is that a runaway build will keep the agent busy longer than you’d like.  
- **Use production mode and parallelism**: Pass `--mode production` and set `parallelism` in webpack to use all CPU cores. This reduces the build duration but can increase memory usage.  
- **Switch to a managed agent**: If you don’t have a dedicated DevOps person, Azure’s hosted agents are pre‑configured with enough resources for most React builds. The cost is a per‑minute fee, but you avoid VM maintenance.  

### 4. Production Example  
1. **Check current specs**: The agent is a B2s (2 vCPU, 4 GB RAM).  
2. **Scale up**: Spin a D4s (4 vCPU, 16 GB RAM) in the same region.  
3. **Add caching**: In the pipeline YAML, add a cache task before `npm install` to restore `node_modules` and after `npm run build` to save the `build/` folder.  
4. **Increase timeout**: Set `timeoutInMinutes: 120` on the build job.  
5. **Run the build**: `npm run build -- --mode production`.  
6. **Monitor**: Watch the agent’s resource usage in Azure Monitor; you should see CPU near 80 % and memory under 10 GB.  
7. **Iterate**: If the build still stalls, add a swap file or switch to a larger VM.  

With these steps, the build should finish well under the new timeout, and subsequent runs will be fast thanks to caching.
