### 3. Deep Dive & Strategy  

**FAISS (Dense‑Vector Retrieval)**  
- **Pros:** Captures semantic similarity, handles synonyms, paraphrases, and contextual meaning; excellent for “intent‑level” queries.  
- **Cons:** Loses precision on exact term matches, rare identifiers (part numbers, model codes), and short queries where the embedding may be noisy; index size grows with vector dimensionality; requires GPU/CPU‑intensive distance calculations.

**BM25 (Lexical Retrieval)**  
- **Pros:** Deterministic term‑frequency based scoring, perfect for exact‑match entities, dates, acronyms, and short keyword queries; very fast with an inverted index; low memory footprint.  
- **Cons:** No semantic understanding—fails on synonyms, re‑phrasings, or queries that don’t share exact tokens with the document.

**Why Not Dense‑Only?**  
In the capstone, pure FAISS performed well on conceptual questions (“How does Llama perform?”) but **failed completely** on queries that required exact identifiers (e.g., “MPT‑7B specs” or “order #12345”). Users reported missing critical facts, which is unacceptable for a knowledge‑base chatbot. Adding BM25 rescued those precision‑critical cases while preserving the semantic coverage of FAISS.

**Hybrid Architecture (EnsembleRetriever)**  
1. **Dual Indexing:** Each chunk is indexed simultaneously in an in‑memory FAISS index (using BGE embeddings) and a BM25 inverted index (via LangChain’s `BM25Retriever`).  
2. **Parallel Retrieval:** For every user query, both retrievers fetch their top‑k (e.g., 10) candidates.  
3. **Score Normalization:** FAISS distances are converted to similarity scores (`1 / (1 + distance)`) and BM25 scores are min‑max scaled to `[0,1]`.  
4. **Weighted Fusion:** Final relevance = `α * score_FAISS + (1‑α) * score_BM25`.  
5. **Reranking (optional):** The fused pool is re‑sorted and the top‑N are passed to the LLM.

**Calibrating the Weight (α)**  
- **Grid‑Search on Validation Set:** Ran a small labeled QA set (≈200 queries) with ground‑truth relevance judgments. Tested α values from 0.1 to 0.9 in 0.1 increments, measuring MAP@10 and nDCG.  
- **Result:** Best trade‑off at **α ≈ 0.7** (70 % semantic, 30 % lexical). This gave a 15 % lift in exact‑entity recall while keeping conceptual recall high.  
- **Dynamic Adjustment (later iteration):** Added a heuristic that boosts BM25 weight when the query contains ≥2 tokens that match a known identifier pattern (regex for part numbers, dates). This adaptive α further reduced false negatives on rare terms.

Those failures broke user trust because the bot would answer conceptually but drop critical factual details. The alternative, a pure lexical engine like BM25, would retrieve those exact matches perfectly but would ignore the broader meaning of a query, leading to low recall on open‑ended questions. The core decision, therefore, was to combine the two—leveraging FAISS for semantic breadth and BM25 for precise recall—rather than forcing a compromise with either alone.

### **2. Technical Implementation & Mechanics** ###
I built an **EnsembleRetriever** in LangChain that indexes every knowledge‑base chunk twice: once as a 768‑dimensional embedding stored in an in‑memory FAISS index, and once as a tokenized document in a BM25 inverted index. At query time the system runs both retrievers in parallel, each returning its top‑k candidates. Because the scoring scales differ, I first normalize them—FAISS distances are transformed into similarity scores (`1 / (1 + distance)`) and BM25 scores are min‑max scaled to a 0‑1 range. The final relevance score for each candidate is a weighted sum:

`final_score = α × score_FAISS + (1 − α) × score_BM25`

Choosing α required experimentation. I assembled a modest validation set covering both semantic and exact‑match queries, then performed a grid search over α values from 0.1 to 0.9. The sweet spot settled around **α ≈ 0.7**, meaning the system leans 70 % on semantic similarity while still giving 30 % weight to keyword precision. To handle edge cases—queries that contain recognizable identifiers—I added a lightweight heuristic that bumps the BM25 contribution when the query matches a regex pattern for part numbers or dates, ensuring those rare terms are never drowned out.
