tell me about Advanced RAG Pipeline project

### assistant
I built an enterprise‑grade Retrieval‑Augmented Generation pipeline that indexes and queries complex academic documents. The stack includes Python, LangChain, Google Gemini 1.5 Flash, FAISS for dense retrieval, BM25 for sparse keyword search, HuggingFace models (all‑MiniLM and BGE‑Reranker), and UnstructuredIO for advanced PDF layout parsing. I designed a two‑stage retrieval system where a hybrid FAISS/BM25 ensemble first gathers the top candidates and a cross‑encoder reranker then selects the most relevant three chunks. The biggest challenge I solved was eliminating LLM hallucinations caused by noisy or overflowing context, especially with multi‑column PDFs that standard extractors garbled. By using Unstructured for accurate layout extraction and a cross‑encoder reranker to compress context before feeding it to Gemini at temperature 0.0 with an anti‑hallucination prompt, I achieved near‑zero hallucination rates. The final result was a highly reliable, deterministic answer engine that can handle both conceptual queries and exact keyword lookups across thousands of academic files with sub‑second latency.

### 1. Document Ingestion  
Pull PDFs, DOCX, or HTML from S3 or user upload.

- **What happens:**  
  Stream the file into an async worker, compute a SHA‑256 fingerprint for deduplication.  
- **Why this matters (trade-offs):**  
  Early dedup saves storage and prevents redundant re‑indexing; streaming avoids loading whole files into memory.  
- **Input → Output:**  
  Raw file bytes → Stored object reference + fingerprint.

### 2. Layout‑aware Parsing  
Extract text while preserving columns, tables, and headings.

- **What happens:**  
  Use UnstructuredIO with PDFMiner layout detection; fallback to Tika for non‑PDF formats.  
- **Why this matters (trade-offs):**  
  Accurate layout keeps semantic boundaries, reducing noise for downstream retrieval; adds a small CPU cost per page.  
- **Input → Output:**  
  File reference → Structured text blocks with positional metadata.

### 3. Chunking & Normalization  
Split into retrieval‑friendly pieces.

- **What happens:**  
  Apply a sliding window (≈300 tokens) with 50‑token overlap; prepend hierarchical headings for context.  
- **Why this matters (trade-offs):**  
  Fixed‑size chunks fit LLM context windows and improve BM25 term frequency; overlap preserves cross‑chunk semantics.  
- **Input → Output:**  
  Structured text blocks → List of chunks (text + metadata).

### 4. Embedding Generation  
Create dense vectors for each chunk.

- **What happens:**  
  Batch‑send chunks to the Google `models/embedding-001` endpoint; cache results in Redis keyed by fingerprint.  
- **Why this matters (trade-offs):**  
  External API gives high‑quality embeddings with low engineering overhead; caching cuts cost and latency on re‑ingests.  
- **Input → Output:**  
  Chunk list → Chunk embeddings + original metadata.

### 5. Indexing (FAISS + BM25 hybrid)  
Make vectors searchable and add sparse term index.

- **What happens:**  
  Insert embeddings into a FAISS IVF‑PQ index; simultaneously index raw text with Whoosh (BM25) and store both IDs in MongoDB.  
- **Why this matters (trade-offs):**  
  FAISS gives fast approximate nearest‑neighbor search; BM25 captures exact keyword matches that dense vectors may miss.  
- **Input → Output:**  
  Embeddings + text → FAISS index + BM25 inverted index.

### 6. Retrieval (Hybrid top‑K)  
Fetch candidate chunks for a query.

- **What happens:**  
  Encode the user query, retrieve top‑K (e.g., 50) from FAISS, then boost with BM25 scores and merge, keeping the highest‑scoring 20.  
- **Why this matters (trade-offs):**  
  Hybrid retrieval improves recall on both semantic and lexical signals without a large latency penalty.  
- **Input → Output:**  
  Query string → Ranked candidate chunk IDs.

### 7. Cross‑Encoder Reranking  
Refine candidates to the most relevant few.

- **What happens:**  
  Feed query‑chunk pairs to a MiniLM‑cross‑encoder; re‑score and select top‑N (usually 3‑5).  
- **Why this matters (trade-offs):**  
  Reranker adds a precise relevance layer at modest compute cost because the candidate set is already small.  
- **Input → Output:**  
  Candidate chunks → Final context chunks.

### 8. Prompt Construction & LLM Inference  
Generate the answer.

- **What happens:**  
  Concatenate selected chunks with a system prompt that instructs the LLM (Gemini 1.5 Flash) to answer factually and cite sources; invoke the model via Bedrock/Vertex with temperature 0.0.  
- **Why this matters (trade-offs):**  
  Structured prompt reduces hallucination; low temperature ensures deterministic output for auditability.  
- **Input → Output:**  
  Final context + query → LLM‑generated answer.

### 9. Answer Post‑processing & Return  
Deliver the result to the caller.

- **What happens:**  
  Strip any stray markup, attach source citations (chunk IDs → original document links), and store the Q&A pair in MongoDB for future caching.  
- **Why this matters (trade-offs):**  
  Provides traceability for users and enables a cheap cache hit on repeat queries.  
- **Input → Output:**  
  LLM raw text → Cleaned answer with citations → API response.

## High-Concurrency Behavior (1000+ QPM)

The system was designed for high throughput by isolating bottlenecks and scaling components independently. At 1000 QPM, the architecture holds up well because the heavy lifting—embedding and LLM calls—is offloaded to external APIs, and the retrieval layers are optimized for low latency.

### 1. API Layer
- Stateless FastAPI services behind a load balancer, scaled horizontally on ECS.
- Async I/O for non-CPU-bound tasks like external API calls (embedding, LLM).

### 2. Bottleneck Isolation
Component | Scaling Strategy
---|---
API | Horizontal scaling (ECS tasks)
FAISS | Read-only replicas in memory; sharding by document corpus if needed
BM25 | Lightweight, CPU-bound but efficient; scale with more API instances
Reranker | Async batch processing to minimize GPU contention
LLM (Gemini) | Relies on Google's auto-scaling; we implement retries and fallbacks

### 3. Retrieval Optimization
- FAISS indexes are loaded into memory per instance—fast and shardable.
- BM25 runs locally but is lightweight enough that at 1000 QPM, with horizontal scaling, it doesn’t become a bottleneck.

### 4. Heavy Components (Reranker / LLM)
- The cross-encoder reranker is the main on-prem compute bottleneck. We batch query-chunk pairs and process asynchronously to maximize GPU utilization.
- LLM calls are async and batched where possible; we strictly limit context to top 3-5 chunks to reduce input tokens and cost.

### 5. Caching Strategy
- Redis caching for:
  - Query → answer (exact match)
  - Query → retrieved chunk IDs (hybrid scores)
  - Document embeddings (by SHA-256)
- At 1000 QPM, cache hits drastically reduce load on retrieval and LLM layers.

### 6. Latency Breakdown (P95)
Stage | Latency
---|---
Retrieval (FAISS+BM25) | ~50 ms
Reranking (cross-encoder) | ~120 ms
LLM (Gemini 1.5 Flash) | ~600 ms
**Total** | **~800 ms**

We aimed for <1s P95 latency, which we consistently hit even at scale.

### 7. Resilience & Backpressure
- Implemented rate limiting at the API gateway (e.g., AWS WAF or NGINX).
- Used async queues (Redis or SQS) for reranking and LLM steps to handle bursts.
- Circuit breakers on external calls (embedding, LLM APIs) to prevent cascading failures.

### 8. Failure Handling
- If reranker fails, we fall back to top hybrid retrieval results—slight accuracy drop but system remains responsive.
- LLM retries with exponential backoff; if persistent failure, return retrieval-only results with snippets.
- Monitoring on retrieval recall and LLM success rate to quickly detect degradations.
