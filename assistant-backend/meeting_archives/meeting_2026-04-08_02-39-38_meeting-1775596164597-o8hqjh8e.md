## Latency Breakdown (End-to-End Pipeline)
- Average latency: ~180 ms per request  
- Target: <1 s P95 under peak load  

### 1. API Layer (NGINX + Node.js/Express)
- Deployed behind an AWS Application Load Balancer; TLS termination at ALB  
- Latency: ~40 ms  
- Used keep‑alive connections and gzip compression  

### 2. Business Logic & Caching
- Node.js services query Redis for session & product‑lookup data  
- Latency: ~30 ms (cache hit) / ~120 ms (cache miss)  
- Cache‑aside pattern with 5‑minute TTL for catalog data  

### 3. Relational Store (PostgreSQL on RDS)
- Read‑replica for catalog reads; primary for writes (orders, inventory)  
- Latency: ~70 ms for read‑replica queries, ~150 ms for write transactions  

### 4. External Services (Payment Gateway, SMS)
- Asynchronous webhook handling via SQS; immediate response returns 202  
- Latency: not in request‑path, <10 ms overhead for enqueue  

### 5. Bottleneck Analysis
- Cache miss latency dominates when hot keys evict (~30 % of total)  
- Write path to primary DB spikes during flash‑sale windows (~20 % of total)  

### 6. Optimization Levers
- Warm‑up Redis with LFU eviction; increase instance memory to raise hit ratio → cut cache‑miss latency by ~50 %  
- Introduce write‑through queue (Kafka) to batch inventory updates; offload to background worker → reduce write latency to ~80 ms  
- Auto‑scale EC2 instances via target tracking on ALB request count; add spot instances for cost‑effective burst capacity  

### 7. Trade‑offs
- Larger Redis memory raises cost but improves hit rate and user‑perceived speed  
- Batching writes introduces slight inventory lag (<2 s) acceptable for retail rushes but requires eventual consistency handling

The stack combined Python, LangChain, Google Gemini 1.5 Flash, FAISS for dense vectors, BM25 for sparse keyword search, HuggingFace models like all‑MiniLM and the BGE‑Reranker, and UnstructuredIO for advanced PDF layout parsing. To tackle the biggest hurdle—preventing LLM hallucinations caused by noisy or irrelevant retrieved chunks—I designed a two‑stage retrieval system that first merges FAISS and BM25 results and then reranks the top ten candidates with a cross‑encoder reranker. I also solved the data‑scraping problem of multi‑column PDFs by replacing basic loaders with Unstructured’s layout‑aware parser, preserving tables and headings. The final pipeline feeds only the three highest‑scoring chunks into Gemini 1.5 Flash at temperature 0.0 with a strict anti‑hallucination prompt. The result was near‑zero hallucinations and highly accurate, fact‑grounded answers across both conceptual and exact‑match queries, making the prototype ready for mission‑critical enterprise use.
