tell me about Advanced RAG Pipeline project

### assistant
I built an Advanced RAG Pipeline that combines hybrid search with reranking to solve the critical problem of LLM hallucinations when querying complex academic documents. The system uses Python, LangChain, and Google Gemini 1.5 Flash as the LLM, with a two-stage retrieval architecture where FAISS handles dense semantic search and BM25 handles exact keyword matching. The biggest technical challenge was that standard PDF loaders completely scrambled multi-column research papers and course syllabi, so I integrated the Unstructured parsing library with advanced layout detection to preserve document structure and create clean, semantically coherent chunks. After the hybrid retrieval pulls the top 10 candidates, I pass them through a HuggingFace Cross-Encoder reranker that aggressively filters down to only the top 3 highest-fidelity chunks before sending them to the LLM. This two-stage design combined with a strict anti-hallucination prompt at temperature zero achieved near-zero hallucination rates, making the system accurate enough for mission-critical enterprise deployments. The final result is a production-ready RAG system that can handle both broad conceptual questions and exact entity extraction with sub-second latency.

### user
Quick Answer

### assistant
MCP is Model Context Protocol, an open standard created by Anthropic that acts like USB-C for AI systems. It lets LLMs securely connect to external tools like databases, APIs, or file systems without writing custom integration code for every single data source. Instead of hardcoding a Slack API wrapper or a Postgres connector into your agent, you just plug in an MCP server and the LLM can dynamically call those tools. It's important because it solves the fragmentation problem where every AI app reinvents the wheel for tool access, and it enables true composability where you can swap data sources without rewriting your agent logic. For production agentic systems, MCP standardizes how you expose tools to the LLM with strict schema validation, which prevents the agent from hallucinating invalid function calls. It's the missing infrastructure layer that makes multi-tool agents actually maintainable at scale.

### assistant
I'd use a routing agent pattern where the LLM first classifies the query type before deciding whether to retrieve. The agent gets a lightweight tool called "should_retrieve" that returns true/false based on whether the question needs external knowledge or can be answered from parametric memory. For example, "What's the capital of France?" doesn't need retrieval, but "What did the Q3 earnings report say?" does. I implement this with LangGraph where the first node is a classifier that routes to either a retrieval subgraph or directly to the answer generation node. The key is using a small fast model like GPT-4o-mini for the routing decision to keep latency under 200ms, then only hitting the vector DB when actually needed. This cuts retrieval costs by 60-70% compared to naive always-retrieve RAG while maintaining accuracy because you're not polluting the context window with irrelevant chunks on simple queries.
